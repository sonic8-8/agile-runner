# 현재 활성 단계

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 현재 단계 문서다.
검증 기준과 작업 문서는 이 문서의 활성 범위를 기준으로 작성하며, GitHub 이슈는 각 작업을 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 단계
### ID
SPEC-0032

### 이름
운영용 조회 응답 반복 검증 스크립트 유지 기준 정리

### 목표
- `SPEC-0031`에서 정리한 적용 가이드와 대표 검증 스크립트가 이후 변경에도 함께 유지되게 기준을 정리한다.
- 스크립트, 가이드 문서, 기준 파일, 테스트, 출력 파일 이름이 어긋나지 않도록 같이 갱신해야 하는 대상을 문서로 고정한다.
- 출력 파일 이름이 바뀌거나 기준 경로가 달라질 때 어디서 먼저 어긋남을 감지해야 하는지와 유지 보수 체크리스트를 정리한다.

### 대상 문제
- `SPEC-0031`까지 오면서 적용 순서와 출력 파일 읽는 기준은 정리됐지만, 이후 변경에서 무엇을 같이 고쳐야 하는지는 문서에 한 번에 모여 있지 않다.
- 스크립트 이름, 출력 파일 이름, 가이드 링크, 기준 파일 경로, 테스트 기대값이 따로 바뀌면 문서와 실제 검증 절차가 쉽게 어긋날 수 있다.
- 지금 상태에서는 새 작업자가 스크립트 유지 보수 범위를 판단할 때 회고와 가이드를 여러 군데 오가며 읽어야 한다.

### 범위
- 운영용 조회 응답 반복 검증 스크립트 유지 기준을 정리한다.
- 아래 내용을 최소 범위로 포함한다.
  - 유지 기준 시작 전 기존 근거와 안전망 확인
  - 스크립트 변경 시 같이 갱신해야 하는 문서와 기준 파일 정리
  - 출력 파일 이름 변경과 문서 어긋남 감지 기준 정리
  - 유지 보수 체크리스트와 단계 마감 근거 정리
- 현재 운영용 조회 응답 반복 검증 세트에 적용한다.
  - `prepare-seed.sh`
  - `run-rerun.sh`
  - `run-retry.sh`
  - `collect-evidence.sh`
  - `manual-rerun-response-seed-command-guide.md`
  - `manual-rerun-response-guide.md`
  - 관련 기준 파일과 자동 검증 테스트

### 비대상
- 새 스크립트 구현
- 기존 스크립트 이름 변경
- 운영용 API 계약 변경
- 실제 앱/H2 대표 검증 재실행
- 관리자 UI

### 외부 계약
- 기존 재실행, 재시도, 단건 조회, 목록 조회, 이력 조회, 관리자 조치 경로와 HTTP 상태 계약은 유지한다.
- 기존 스크립트 이름과 현재 출력 파일 이름은 이번 단계에서 바꾸지 않는다.
- 이번 단계는 유지 기준 정리 작업이지, 대표 검증 절차를 새로 늘리거나 운영 API 계약을 고치는 단계가 아니다.

### 핵심 시나리오
1. 유지 기준 시작 전 기존 근거와 안전망 확인
   - `SPEC-0031` 마감 근거와 현재 가이드, 테스트가 유지 기준 정리를 시작하기에 충분한지 먼저 확인한다.
   - 부족한 경우만 최소 문서 보강을 검토한다.
2. 변경 시 함께 갱신해야 하는 문서와 기준 파일 정리
   - 스크립트, 가이드, 기준 파일, 관련 테스트 중 무엇을 같이 봐야 하는지 한 번에 찾게 정리한다.
3. 출력 파일 이름 변경과 문서 어긋남 감지 기준 정리
   - 출력 파일 이름이 바뀔 때 어떤 문서와 테스트에서 가장 먼저 어긋남을 잡아야 하는지 정리한다.
4. 유지 보수 체크리스트와 단계 마감
   - 이후 변경 시 따라야 하는 체크리스트와 문서 갱신 순서를 정리하고 단계 마감 근거를 남긴다.

### 작업 분해 기준
- `TASK-0001` 유지 기준 시작 전 기존 근거와 안전망 확인
- `TASK-0002` 변경 시 함께 갱신해야 하는 문서와 기준 파일 정리
- `TASK-0003` 출력 파일 이름 변경과 문서 어긋남 감지 기준 정리
- `TASK-0004` 유지 보수 체크리스트와 단계 마감

### 연결 검증 기준
- `manual-rerun-script-maintenance-safety-net-preserved`
- `manual-rerun-script-maintenance-sync-boundary-documented`
- `manual-rerun-script-maintenance-drift-detection-documented`
- `manual-rerun-script-maintenance-checklist-closeout-documented`

### 필수 테스트 시나리오
- 가이드와 기준 파일 정리 동안 기존 재실행, 재시도, 단건 조회, 목록 조회, 이력 조회, 관리자 조치 자동 검증은 유지된다.
- 운영자가 스크립트나 출력 파일 이름을 바꿀 때 어떤 문서와 테스트를 같이 봐야 하는지 문서만으로 다시 따라갈 수 있어야 한다.
- 유지 보수 체크리스트를 보고 이후 변경의 점검 순서를 다시 구성할 수 있어야 한다.

## 후속 단계 후보
### ID
SPEC-0033

### 이름
운영용 조회 응답 반복 검증 스크립트 오류 대응 기준 정리

### 시작 조건
- `현재 활성 단계`가 완료되고, 유지 기준과 점검 체크리스트가 정리된 뒤 시작한다.

### 목표
- 반복 검증 스크립트 실행 중 멈춤, 출력 누락, H2 잠금 같은 오류를 어떤 순서로 해석하고 분리할지 정리한다.

### 후속 변경 범위
- 스크립트 종료 코드 해석 기준 재정리
- 출력 파일 누락 시 점검 순서 정리
- H2 잠금과 코드 오류 분리 기준 정리

### 후속 변경 비대상
- 사용자용 UI
- 장기 저장소 도입

### 후속 검증 방향
- 현재 구현 결과를 바탕으로 스크립트 실패 시 운영자가 어디서부터 다시 확인해야 하는지 기준을 더 정리한다.
