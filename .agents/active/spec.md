# 현재 활성 단계

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 현재 단계 문서다.
검증 기준과 작업 문서는 이 문서의 활성 범위를 기준으로 작성하며, GitHub 이슈는 각 작업을 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 단계
### ID
SPEC-0029

### 이름
운영용 조회 응답 준비 데이터 반복 검증 스크립트 초안 구현 검토

### 목표
- 운영용 조회 응답 대표 검증 절차에서 실제 스크립트 초안을 만든다면 어떤 파일 구조와 입력/출력 기준으로 시작할지 검토한다.
- 현재 보조 명령 가이드 기준 절차에서 어떤 명령 묶음을 초안 파일 후보 범위로 삼고, 어떤 단계는 계속 수동 확인으로 남길지 정리한다.
- 스크립트 초안 파일 후보의 입력/출력, 인계 지점, 비대상을 문서로 고정해 다음 단계 구현 여부를 판단할 근거를 남긴다.

### 대상 문제
- SPEC-0028에서 초안 범위, 중단 조건, 수동 확인 경계는 정리했지만, 실제 초안 파일을 만든다면 어떤 입력 인자, 출력 파일, 종료 지점을 한 번에 가져갈지는 아직 문서에 없다.
- 초안 구현 검토 없이 바로 스크립트 파일을 만들면 문서 기반 절차와 스크립트 기반 절차가 어긋나거나, 수동 확인 단계까지 잘못 끌어올 위험이 있다.
- 반대로 구현 검토를 더 미루면 실제 반복 비용을 얼마나 줄일 수 있는지 계속 문서 수준에서만 맴돌게 된다.

### 범위
- 운영용 조회 응답 대표 검증 절차를 실제 스크립트 초안 파일로 옮긴다면 어떤 구조로 시작할지 검토한다.
- 아래 내용을 최소 범위로 포함한다.
  - 초안 스크립트 파일 후보 위치와 역할 정리
  - 초안 입력 인자와 출력 파일 묶음 정리
  - 초안이 직접 다루는 명령 범위와 수동 확인 인계 지점 정리
  - 초안 종료 코드와 실패 종료 흐름 검토
  - 다음 단계 구현으로 넘어갈지 판단하는 마감 근거
- 현재 운영용 조회 응답 세트에 적용한다.
  - `rerun`
  - `retry`
  - 단건 조회
  - 목록 조회
  - 이력 조회
  - 관리자 조치 응답

### 비대상
- 실제 스크립트 파일 추가
- 새 경로 추가
- 응답 필드 추가 또는 이름 변경
- 준비 데이터 SQL 자체 구조 변경
- 운영자 UI

### 외부 계약
- 기존 rerun/retry/단건 조회/목록 조회/이력 조회/관리자 조치 경로와 HTTP 상태 계약은 유지한다.
- 기존 응답 필드 의미는 바꾸지 않는다.
- 이번 단계는 대표 검증 스크립트 초안 파일 후보를 검토하는 작업이며, 관리 API 계약 자체를 바꾸는 작업이 아니다.

### 핵심 시나리오
1. 초안 구현 검토 시작 전 기존 판단과 근거 확인
   - `SPEC-0028` 단계 요약 문서와 현재 가이드 문서가 스크립트 초안 구현 검토 단계의 시작 근거로 충분한지 먼저 확인한다.
   - 부족한 경우만 최소 문서 보강을 검토한다.
2. 초안 파일 구조와 입력/출력 계약 정리
   - 초안 스크립트를 만든다면 어떤 파일 위치와 어떤 입력 인자 묶음으로 시작할지 정리한다.
   - 출력 파일과 기존 응답/H2 조회 근거 파일을 어떤 이름으로 유지할지도 함께 정리한다.
3. 실패 종료 흐름과 수동 확인 인계 지점 정리
   - 어떤 실패에서 초안이 종료돼야 하는지, 종료 상태를 나눌 필요가 있는지, 어떤 시점에서 사람에게 판단이 넘어가는지 정리한다.
   - 응답 값 해석, H2 의미 해석, 회고 작성은 계속 수동 단계로 유지한다.
4. 초안 구현 검토 판단과 다음 단계 마감
   - 현재는 초안 구현 검토로 충분한지, 아니면 후속 단계에서 실제 스크립트 구현으로 넘어가야 하는지 판단 근거를 남긴다.
   - 대상 테스트와 전체 테스트를 다시 돌려 문서 정리로 기존 검증 흐름이 흔들리지 않았는지 확인한다.

### 작업 분해 기준
- `TASK-0001` 초안 구현 검토 시작 전 기존 판단과 근거 확인
- `TASK-0002` 초안 파일 구조와 입력/출력 계약 정리
- `TASK-0003` 실패 종료 흐름과 수동 확인 인계 지점 정리
- `TASK-0004` 초안 구현 검토 판단과 단계 마감

### 연결 검증 기준
- `manual-rerun-script-implementation-review-safety-net-preserved`
- `manual-rerun-script-implementation-draft-structure-documented`
- `manual-rerun-script-implementation-handoff-documented`
- `manual-rerun-script-implementation-review-closeout-completed`

### 필수 테스트 시나리오
- 스크립트 초안 구현 검토 문서를 정리하는 동안 기존 rerun/retry/단건 조회/목록 조회/이력 조회/관리자 조치 자동 검증은 유지된다.
- 현재 보조 명령 가이드와 회고만으로 초안 파일 후보 구조와 인계 지점을 다시 재구성할 수 있어야 한다.
- 어떤 파일 구조와 입력/출력 계약으로 시작할지, 어떤 실패 종료 흐름과 수동 확인 단계는 계속 남기는지 문서에서 구분되어야 한다.

## 후속 단계 후보
### ID
SPEC-0030

### 이름
운영용 조회 응답 준비 데이터 반복 검증 스크립트 초안 구현

### 시작 조건
- `현재 활성 단계`가 완료되고, 대표 검증 보조 명령이 문서화된 뒤 시작한다.

### 목표
- `현재 활성 단계`에서 초안 파일 구조와 인계 지점이 충분히 좁혀졌다고 판단되면, 실제 스크립트 초안 구현으로 넘어간다.

### 후속 변경 범위
- 보조 스크립트 초안 파일 추가
- 입력/출력 파일과 실행 경계 실제 반영
- 문서 기반 절차와 초안 기반 절차의 실제 전환 기준 적용

### 후속 변경 비대상
- 사용자용 UI
- 장기 저장소 도입

### 후속 검증 방향
- 현재 검토 결과를 바탕으로 실제 초안 파일이 필요한지 판단하고, 필요하면 최소 구현 범위를 정한다.
