# 현재 활성 Spec

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 `Spec` 문서다.
`ValidationCriteria`와 `Task`는 이 문서의 활성 spec을 기준으로 작성하며, GitHub Issue는 각 `Task`를 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 Spec
### ID
SPEC-0007

### 이름
선택 실행 기능 기반 마련

### 목표
- 수동 재실행 시 전체 PR이 아니라 선택한 파일 경로만 대상으로 리뷰를 다시 실행할 수 있는 기반을 만든다.
- 선택 실행 입력은 현재 `manual rerun` 진입점에서 명시적으로 다룰 수 있어야 한다.
- runtime evidence에 어떤 선택 조건으로 실행했는지 남길 수 있는 기반을 만든다.

### 대상 문제
- 현재 수동 재실행은 PR 전체를 다시 실행하므로, 일부 파일만 다시 보고 싶은 경우에도 전체 리뷰와 전체 코멘트 경로를 다시 타게 된다.
- 실행 제어 모드는 생겼지만, 실행 범위를 줄이는 선택 조건은 아직 입력 모델과 실행 근거에 없다.
- 운영 관점에서 특정 파일만 대상으로 재실행했는지, 전체 PR을 다시 실행했는지 구분하기 어렵다.

### 범위
- 선택 실행 조건은 우선 `파일 경로 목록`으로 한정한다.
- 수동 재실행 요청은 선택 파일 경로 목록을 명시적으로 전달할 수 있어야 한다.
- 리뷰 생성 입력과 경로 기반 inline comment 대상은 선택한 파일 경로에 해당하는 항목만 유지되도록 정리한다.
- 선택 조건은 runtime evidence에 `선택 실행 여부 + 정렬된 파일 경로 목록 요약 문자열` 형태로 남길 수 있어야 한다.
- 로컬 프로필 실제 앱/H2 기준으로 대표 선택 실행 1건의 실행 근거를 검증한다.

### 비대상
- glob, 정규식, 디렉토리 패턴 같은 복잡한 선택 문법
- line 단위 선택 실행
- label/comment 기반 선택 실행
- 운영 대시보드, 조회 API, 장기 저장소 도입
- 자동 재시도, dead-letter 저장소 도입

### 외부 계약
- `/webhook/github` 엔드포인트의 성공 응답과 조기 종료 계약은 유지한다.
- `POST /reviews/rerun`은 계속 내부/관리자용 기반 진입점으로 유지한다.
- 이번 spec에서는 `POST /reviews/rerun` 요청에 선택 파일 경로 목록을 추가할 수 있어야 한다.
- 선택 조건이 비어 있으면 기존처럼 전체 PR 기준으로 동작해야 한다.
- 선택 파일 경로가 PR diff와 하나도 매칭되지 않아도 기존 성공 응답 계약은 유지하고, 리뷰 입력과 경로 기반 inline comment 대상은 빈 상태로 처리한다.
- 수동 재실행 성공 응답의 최소 계약 `executionKey`, `executionControlMode`, `writePerformed`는 유지한다.

### 핵심 시나리오
1. 선택 실행 안전망 고정
   - 기존 webhook 계약과 manual rerun 계약이 깨지지 않는다는 점을 먼저 테스트로 고정한다.
   - 선택 파일 경로를 아직 적용하지 않은 전체 경로와, 선택 조건이 비어 있는 전체 실행 경로를 구분해 기준을 세운다.
2. 선택 실행 입력 모델 도입
   - manual rerun 요청과 service request에서 파일 경로 목록을 명시적으로 다룬다.
   - 선택 파일 경로가 비어 있을 때는 전체 실행으로 해석한다.
3. 선택 경로 기준 리뷰/코멘트 제한
   - 리뷰 생성 입력과 경로 기반 inline comment 대상은 선택한 파일 경로에 해당하는 항목만 남긴다.
   - `NORMAL`, `DRY_RUN` 실행 제어는 기존 규칙을 그대로 유지한다.
4. 선택 실행 근거 적재와 실제 검증
   - runtime evidence에 선택 실행 여부와 정렬된 파일 경로 목록 요약 문자열을 남긴다.
   - 로컬 프로필 실제 앱/H2 기준으로 대표 선택 실행 1건의 실행 근거를 검증한다.

### Task 분해 기준
- `TASK-0001` 선택 실행 안전망 고정
- `TASK-0002` 선택 실행 입력 모델 도입
- `TASK-0003` 선택 경로 기준 리뷰/코멘트 제한
- `TASK-0004` 선택 실행 근거 적재와 실제 검증

### 연결될 ValidationCriteria
- `webhook-and-rerun-contract-preserved-during-selective-execution`
- `selection-input-resolved-consistently`
- `selected-paths-limit-review-and-comment-scope`
- `runtime-evidence-records-selection-scope`

### 필수 테스트 시나리오
- 선택 실행 기능을 추가한 뒤에도 `/webhook/github`의 성공 응답과 조기 종료 계약은 유지된다.
- `POST /reviews/rerun`은 선택 파일 경로 목록을 포함한 요청을 일관되게 해석한다.
- 선택 파일 경로가 비어 있으면 기존 전체 실행과 같은 경로를 유지한다.
- 선택 파일 경로가 있으면 리뷰 생성 입력과 경로 기반 inline comment 대상은 해당 경로만 남긴다.
- 선택 파일 경로가 PR diff와 하나도 매칭되지 않으면 리뷰 입력과 경로 기반 inline comment 대상은 빈 상태로 처리되며, 기존 성공 응답 계약은 유지된다.
- 로컬 프로필 실제 앱 기동 시 대표 선택 실행 응답의 `executionKey`를 기준으로 runtime evidence에서 선택 실행 여부와 정렬된 파일 경로 목록 요약 문자열을 확인할 수 있다.

## 후속 Spec 후보
### ID
SPEC-0008

### 이름
재실행 응답 모델 정교화

### 시작 조건
- `현재 활성 Spec`이 완료되고, 선택 실행 입력과 실행 근거가 안정적으로 정리된 뒤 시작한다.

### 목표
- 수동 재실행 응답에서 실패 상태와 사용자 노출 범위를 더 명확히 표현하는 방향을 검토한다.

### 후속 변경 범위
- manual rerun 응답의 실패 상태 표기 검토
- 내부/관리자용 응답 정책 구체화 검토
- 실행 결과 조회 모델과의 연결 검토

### 후속 변경 비대상
- 운영 대시보드 구축
- 장기 저장소 도입
- 자동 재시도 정책

### 후속 검증 방향
- 수동 재실행 응답이 runtime evidence와 모순되지 않는다.
- 선택 실행 기능과 실패 상태 모델이 함께 있어도 기존 webhook 계약은 유지된다.
