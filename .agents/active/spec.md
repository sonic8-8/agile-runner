# 현재 활성 Spec

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 `Spec` 문서다.
`ValidationCriteria`와 `Task`는 이 문서의 활성 spec을 기준으로 작성하며, GitHub Issue는 각 `Task`를 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 Spec
### ID
SPEC-0024

### 이름
운영용 조회 응답 준비 데이터 적용 절차 정리

### 목표
- 운영용 조회 응답 대표 검증 전에 준비 데이터를 어떤 순서로 적용할지 새 작업자도 반복 가능하게 정리한다.
- 준비 데이터 적용 전에 schema와 enum 값을 어떤 순서로 점검할지 고정해 준비 오류 때문에 먼저 실패하는 일을 줄인다.
- 대표 검증 실행 전 체크리스트를 문서로 고정해, 준비 데이터 규칙과 실제 적용 절차를 분리해서 이해할 수 있게 한다.
- 예: `schema 확인 -> enum 값 확인 -> retry 원본 실행 준비 데이터 적용 -> 관리자 조치 이력 준비 데이터 적용 -> 앱 기동 -> 대표 요청 실행 -> 앱 종료 -> 실행 근거 확인 SQL 실행` 같은 기본 흐름을 문서에서 바로 읽을 수 있게 한다.

### 대상 문제
- 준비 데이터 규칙과 파일 구조는 정리됐지만, 실제로 local H2에 어떤 순서로 적용하고 어떤 항목을 먼저 확인해야 하는지 절차 문서가 아직 없다.
- 같은 준비 데이터를 두고도 작업자마다 적용 순서가 달라지면 대표 검증 실패 원인이 제품 버그인지 준비 절차 문제인지 다시 헷갈릴 수 있다.
- schema나 enum 값을 확인하지 않고 준비 데이터를 넣으면 준비 오류가 먼저 실패로 드러날 수 있는데, 지금은 그 점검 순서와 체크리스트가 문서로 고정돼 있지 않다.

### 범위
- 운영용 조회 응답 대표 검증 전에 준비 데이터를 적용하는 절차를 문서로 정리한다.
- 아래 내용을 최소 범위로 포함한다.
  - 준비 데이터 적용 순서
  - 준비 데이터 적용 전에 schema와 enum 값을 어떤 순서로 확인하는지
  - 대표 검증 전에 확인할 체크리스트
  - 준비 데이터 파일, 문서용 기준 파일, 대표 검증 결과를 순서상 언제 구분해서 다루는지
- 현재 운영용 조회 응답 세트에 적용한다.
  - `rerun`
  - `retry`
  - `query`
  - `list`
  - `history`
  - `action`
- 현재 대표 검증에서 실제로 필요했던 retry 원본 실행 준비 데이터, 관리자 조치 이력 준비 데이터, list/query 대표 row 준비 경로를 우선 대상으로 삼는다.
- 현재 준비 데이터 가이드와 준비 데이터 파일 구조를 기준으로 실제 적용 순서를 문서와 체크리스트로 정리한다.

### 비대상
- 새 조회 endpoint 추가
- 응답 필드 추가 또는 rename
- 기준 파일 자동 생성 스크립트 도입
- 대표 검증 전체 자동화
- 준비 데이터 이름 규칙과 저장 위치 규칙 재정의
- 사용자용 UI

### 외부 계약
- 기존 rerun/retry/query/list/history/action 경로와 HTTP status 계약은 유지한다.
- 기존 응답 필드 의미는 바꾸지 않는다.
- 이번 spec은 준비 데이터 적용 절차와 점검 체크리스트를 정리하는 작업이며, 운영 API 동작 자체를 바꾸는 작업이 아니다.

### 핵심 시나리오
1. 준비 데이터 적용 절차 안전망 확인
   - 대표 실제 앱/H2 검증에서 준비 데이터 적용 순서가 실제로 필요했던 지점과 기존 회고 근거가 이미 충분한지 먼저 확인한다.
   - 부족한 경우만 최소 문서/테스트 보강을 검토한다.
2. 준비 데이터 적용 순서와 명령 기준 정리
   - 어떤 준비 데이터 파일을 어떤 순서로 적용하는지 정리한다.
   - retry 원본 실행 준비 데이터, 관리자 조치 이력 준비 데이터, 실행 근거 확인 SQL을 어떤 순서로 다루는지 고정한다.
3. schema와 enum 점검 순서, 대표 검증 체크리스트 정리
   - 준비 데이터 적용 전에 schema와 enum 값을 어떤 순서로 확인하는지 적는다.
   - 대표 검증 전에 빠뜨리면 안 되는 점검 항목을 체크리스트로 정리한다.
   - 예: `schema 확인 -> enum 값 확인 -> 준비 데이터 적용 -> 앱 기동` 같은 시작 순서를 문서에 명시한다.
4. 준비 데이터 적용 절차 문서 마감과 정합성 확인
   - 새 작업자가 문서와 준비 데이터 파일만 읽고 대표 검증 준비 절차를 따라갈 수 있는지 점검한다.
   - targeted/full test를 다시 돌려 문서 정리로 기존 자동 검증 흐름이 깨지지 않았는지 확인한다.

### Task 분해 기준
- `TASK-0001` 준비 데이터 적용 절차 안전망 확인
- `TASK-0002` 준비 데이터 적용 순서와 명령 기준 정리
- `TASK-0003` schema와 enum 점검 순서, 대표 검증 체크리스트 정리
- `TASK-0004` 준비 데이터 적용 절차 문서 마감과 정합성 확인

### 연결될 ValidationCriteria
- `manual-rerun-seed-application-safety-net-preserved`
- `manual-rerun-seed-application-order-defined`
- `manual-rerun-seed-precheck-checklist-defined`
- `manual-rerun-seed-application-guide-readiness-verified`

### 필수 테스트 시나리오
- 준비 데이터 적용 절차를 정리하는 동안 기존 rerun/retry/query/list/history/action 자동 검증은 유지된다.
- 새 작업자가 문서만 읽고 준비 데이터 적용 순서와 점검 순서를 이해할 수 있다.
- 대표 실제 앱/H2 검증 전에 확인할 체크리스트가 문서에서 바로 읽힌다.
- 문서 정리 후에도 targeted test와 full test에서 기존 자동 검증 흐름이 유지된다.

## 후속 Spec 후보
### ID
SPEC-0025

### 이름
운영용 조회 응답 준비 데이터 실제 SQL 보강

### 시작 조건
- `현재 활성 Spec`이 완료되고, 준비 데이터 적용 순서와 점검 체크리스트가 문서로 고정된 뒤 시작한다.

### 목표
- 준비 데이터 example.sql 파일에 실제 SQL 구문을 보강하고, 대표 실제 앱/H2 검증과 연결할 최소 실행 가능 상태를 만든다.

### 후속 변경 범위
- 준비 데이터 example.sql 실제 SQL 보강
- 대표 실제 앱/H2 검증과 준비 데이터 파일 연결 보강
- 준비 데이터 파일의 실행 가능성 점검

### 후속 변경 비대상
- 사용자용 UI
- 장기 저장소 도입

### 후속 검증 방향
- 준비 데이터 파일이 예시 이름 규칙을 넘어서 실제 대표 검증 준비에 쓸 수 있는 수준인지 확인한다.
