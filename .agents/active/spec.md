# 현재 활성 단계

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 현재 단계 문서다.
검증 기준과 작업 문서는 이 문서의 활성 범위를 기준으로 작성하며, GitHub 이슈는 각 작업을 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 단계
### ID
SPEC-0031

### 이름
운영용 조회 응답 반복 검증 스크립트 적용 절차와 출력 기준 정리

### 목표
- `SPEC-0030`에서 추가한 초안 파일을 운영자가 더 쉽게 적용하고 유지할 수 있게 정리한다.
- 대표 검증을 다시 수행할 때 어떤 스크립트부터 실행하고, 어떤 출력 파일을 먼저 확인해야 하는지 이미 있는 내용을 더 빨리 찾게 정리한다.
- 요약 파일(`summary.json`), 응답 가이드, 회고, 출력 파일에 이미 흩어져 있는 역할 설명을 다시 묶어 해석 비용을 줄인다.

### 대상 문제
- `SPEC-0030`에서 초안 파일 자체는 생겼지만, 새 작업자가 실제로 적용하려면 보조 명령 가이드, 출력 디렉토리, 요약 파일, 회고 문서를 여러 번 오가며 해석해야 한다.
- 재실행, 재시도 대표 검증에서 어떤 출력 파일을 먼저 열어야 하는지와, 어떤 파일이 최종 판단 근거인지가 문서상 바로 드러나지 않는다.
- 현재 상태가 계속되면 초안 파일은 있어도 적용 순서 실수와 출력 해석 비용이 줄지 않는다.

### 범위
- 운영용 조회 응답 대표 검증 스크립트 초안 파일의 적용 절차와 출력 해석 기준을 정리한다.
- 아래 내용을 최소 범위로 포함한다.
  - 적용 시작 전 확인해야 할 기존 근거와 안전망 정리
  - `prepare-seed.sh`, `run-rerun.sh`, `run-retry.sh`, `collect-evidence.sh` 적용 순서 정리
  - 재실행, 재시도 출력 파일과 대표 검증 결론의 연결 기준 정리
  - 요약 파일(`summary.json`), 응답 가이드, 회고, 출력 파일의 역할 경계 정리
  - 실제 가이드 기준 대표 검증 재수행과 단계 마감 근거 정리
- 현재 운영용 조회 응답 세트에 적용한다.
  - 재실행(`rerun`)
  - 재시도(`retry`)
  - 단건 조회
  - 목록 조회
  - 이력 조회
  - 관리자 조치 응답

### 비대상
- 새 스크립트 파일 추가
- 기존 스크립트 이름 변경
- 응답 필드 추가 또는 이름 변경
- 준비 데이터 SQL 자체 구조 변경
- 운영자 UI

### 외부 계약
- 기존 재실행, 재시도, 단건 조회, 목록 조회, 이력 조회, 관리자 조치 경로와 HTTP 상태 계약은 유지한다.
- 기존 응답 필드 의미와 기존 초안 파일 이름은 바꾸지 않는다.
- 이번 단계는 스크립트 적용 절차와 출력 기준 정리 작업이지, 관리 API 계약이나 스크립트 구현 자체를 크게 바꾸는 작업이 아니다.

### 핵심 시나리오
1. 적용 검증 시작 전 기존 근거와 안전망 확인
   - `SPEC-0030` 단계 요약과 현재 보조 명령 가이드가 `SPEC-0031` 시작 근거로 충분한지 먼저 확인한다.
   - 부족한 경우만 최소 문서 보강을 검토한다.
2. 스크립트 적용 순서와 입력/출력 파일 사용 흐름 정리
   - 이미 보조 명령 가이드에 들어 있는 준비 데이터 적용, 재실행, 재시도, 실행 근거 수집 순서를 빠른 진입 순서로 다시 묶는다.
   - 각 스크립트가 어떤 입력을 받고 어떤 출력 파일을 남기는지 흩어진 설명을 적용 관점으로 다시 배치한다.
3. 대표 검증 결과와 출력 파일 매핑 정리
   - 재실행, 재시도 대표 검증에서 어떤 출력 파일을 어떤 판단 근거와 연결해 읽는지 이미 있는 설명을 한 번에 찾게 다시 배치한다.
   - 요약 파일, 응답 가이드, 회고, 출력 파일의 역할 경계를 한 눈에 읽히는 형태로 정리한다.
4. 스크립트 적용 가이드 대표 검증과 단계 마감
   - 정리된 가이드 기준으로 대표 재실행/재시도 흐름을 다시 따라간다.
   - 출력 파일, 요약 파일, 응답, H2 실행 근거가 같은 실행 키 기준으로 맞는지 확인하고 단계 마감 근거를 남긴다.

### 작업 분해 기준
- `TASK-0001` 적용 검증 시작 전 기존 근거와 안전망 확인
- `TASK-0002` 스크립트 적용 순서와 입력/출력 파일 사용 흐름 정리
- `TASK-0003` 대표 검증 결과와 출력 파일 매핑 정리
- `TASK-0004` 스크립트 적용 가이드 대표 검증과 단계 마감

### 연결 검증 기준
- `manual-rerun-script-application-safety-net-preserved`
- `manual-rerun-script-application-order-documented`
- `manual-rerun-script-application-output-mapping-documented`
- `manual-rerun-script-application-representative-verified`

### 필수 테스트 시나리오
- 가이드 문서 보강 동안 기존 재실행, 재시도, 단건 조회, 목록 조회, 이력 조회, 관리자 조치 자동 검증은 유지된다.
- 운영자가 `prepare-seed.sh`, `run-rerun.sh`, `run-retry.sh`, `collect-evidence.sh`를 어떤 순서로 쓰고, 어떤 출력 파일을 확인해야 하는지 문서만으로 다시 따라갈 수 있어야 한다.
- 대표 재실행/재시도 검증에서 출력 파일, 요약 파일, H2 실행 근거를 같은 실행 키 기준으로 다시 읽을 수 있어야 한다.

## 후속 단계 후보
### ID
SPEC-0032

### 이름
운영용 조회 응답 반복 검증 스크립트 유지 기준 정리

### 시작 조건
- `현재 활성 단계`가 완료되고, 스크립트 적용 기준과 대표 검증 마감 근거가 정리된 뒤 시작한다.

### 목표
- `현재 활성 단계`에서 정리한 적용 기준을 바탕으로, 스크립트와 문서가 이후 변경에도 덜 흔들리게 유지 기준을 정리한다.

### 후속 변경 범위
- 스크립트 변경 시 같이 갱신해야 하는 문서와 기준 파일 정리
- 출력 파일 이름 변경과 문서 어긋남 감지 기준 정리
- 유지 보수 체크리스트 정리

### 후속 변경 비대상
- 사용자용 UI
- 장기 저장소 도입

### 후속 검증 방향
- 현재 구현 결과를 바탕으로 초안 파일 유지와 문서 동기화 기준을 더 정리한다.
