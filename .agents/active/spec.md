# 현재 활성 단계

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 현재 단계 문서다.
검증 기준과 작업 문서는 이 문서의 활성 범위를 기준으로 작성하며, GitHub 이슈는 각 작업을 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 단계
### ID
SPEC-0033

### 이름
운영용 조회 응답 반복 검증 스크립트 오류 대응 기준 정리

### 목표
- 반복 검증 스크립트가 중간에 멈추거나 출력 파일이 빠질 때 운영자가 어디서부터 다시 확인해야 하는지 정리한다.
- H2 잠금, 스크립트 종료 코드, 출력 파일 누락처럼 겉으로 비슷해 보이는 오류를 어떤 순서로 나눠 해석할지 문서로 고정한다.
- 기존 정상 흐름 가이드와 유지 기준 문서 위에 오류 대응 기준을 얹어, 실패 시 되돌아갈 점검 순서를 분명히 한다.

### 대상 문제
- `SPEC-0032`까지 오면서 정상 흐름, 유지 보수 체크리스트, 출력 파일 어긋남 감지 기준은 정리됐지만, 실제 오류가 났을 때는 어디서부터 다시 봐야 하는지가 한 문서에 모여 있지 않다.
- 스크립트 종료 코드가 0이 아니거나, 출력 파일이 없거나, H2 조회가 잠기면 운영자는 코드 문제인지 환경 문제인지 먼저 헷갈릴 수 있다.
- 지금 상태에서는 새 작업자가 오류를 만나면 가이드, 테스트, 회고를 여러 군데 오가며 읽어야 해서 재현 비용과 해석 비용이 크다.

### 범위
- 운영용 조회 응답 반복 검증 스크립트 오류 대응 기준을 정리한다.
- 아래 내용을 최소 범위로 포함한다.
  - 오류 대응 기준 시작 전 기존 근거와 안전망 확인
  - 스크립트 종료 코드와 멈춤 해석 기준 정리
  - 출력 파일 누락 시 점검 순서 정리
  - H2 잠금과 코드 오류 분리 기준, 단계 마감 근거 정리
- 현재 운영용 조회 응답 반복 검증 세트에 적용한다.
  - `prepare-seed.sh`
  - `run-rerun.sh`
  - `run-retry.sh`
  - `collect-evidence.sh`
  - `manual-rerun-response-seed-command-guide.md`
  - `manual-rerun-response-guide.md`
  - 관련 기준 파일과 자동 검증 테스트

### 비대상
- 새 스크립트 구현
- 기존 스크립트 이름 변경
- 운영용 API 계약 변경
- 실제 앱/H2 대표 검증 재실행
- 관리자 UI

### 외부 계약
- 기존 재실행, 재시도, 단건 조회, 목록 조회, 이력 조회, 관리자 조치 경로와 HTTP 상태 계약은 유지한다.
- 기존 스크립트 이름과 현재 출력 파일 이름은 이번 단계에서 바꾸지 않는다.
- 이번 단계는 오류 대응 기준 정리 작업이지, 대표 검증 절차를 새로 늘리거나 운영 API 계약을 고치는 단계가 아니다.

### 핵심 시나리오
1. 오류 대응 기준 시작 전 기존 근거와 안전망 확인
   - `SPEC-0032` 마감 근거와 현재 가이드, 테스트가 오류 대응 기준 정리를 시작하기에 충분한지 먼저 확인한다.
   - 부족한 경우만 최소 문서 보강을 검토한다.
2. 스크립트 종료 코드와 멈춤 해석 기준 정리
   - 스크립트가 어디서 멈췄는지, 종료 코드가 무엇을 뜻하는지, 다음 확인 지점이 어디인지 정리한다.
3. 출력 파일 누락 시 점검 순서 정리
   - 어떤 출력 파일이 없을 때 어떤 스크립트, 테스트, 문서를 먼저 확인해야 하는지 순서를 정리한다.
4. H2 잠금과 코드 오류 분리, 단계 마감
   - H2 잠금과 실제 코드 오류를 어떤 순서로 나눠 확인할지 정리하고 단계 마감 근거를 남긴다.

### 작업 분해 기준
- `TASK-0001` 오류 대응 기준 시작 전 기존 근거와 안전망 확인
- `TASK-0002` 스크립트 종료 코드와 멈춤 해석 기준 정리
- `TASK-0003` 출력 파일 누락 시 점검 순서 정리
- `TASK-0004` H2 잠금과 코드 오류 분리 기준, 단계 마감

### 연결 검증 기준
- `manual-rerun-script-error-safety-net-preserved`
- `manual-rerun-script-stop-code-documented`
- `manual-rerun-script-output-missing-check-order-documented`
- `manual-rerun-script-h2-lock-separation-closeout-documented`

### 필수 테스트 시나리오
- 오류 대응 기준 정리 동안 기존 재실행, 재시도, 단건 조회, 목록 조회, 이력 조회, 관리자 조치 자동 검증은 유지된다.
- 운영자가 스크립트 종료 코드나 출력 파일 누락을 만났을 때 어떤 스크립트, 테스트, 문서를 먼저 봐야 하는지 문서만으로 다시 따라갈 수 있어야 한다.
- H2 잠금과 코드 오류를 섞지 않고 점검 순서를 다시 구성할 수 있어야 한다.

## 후속 단계 후보
### ID
SPEC-0034

### 이름
운영용 조회 응답 반복 검증 스크립트 실패 사례 예시 정리

### 시작 조건
- `현재 활성 단계`가 완료되고, 오류 대응 기준이 정리된 뒤 시작한다.

### 목표
- 반복 검증 스크립트 실패 사례를 실제 예시로 정리해, 운영자가 문서만 보고도 오류 유형별로 같은 판단을 다시 따라갈 수 있게 한다.

### 후속 변경 범위
- 멈춤, 출력 누락, H2 잠금 사례별 예시 추가
- 실패 사례와 대응 기준 연결
- 오류 대응 문서와 마지막 회고 간 참조 보강

### 후속 변경 비대상
- 사용자용 UI
- 장기 저장소 도입

### 후속 검증 방향
- 현재 구현 결과를 바탕으로 운영자가 실패 사례를 실제 문구와 예시 값으로 더 빨리 비교할 수 있게 한다.
