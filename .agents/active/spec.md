# 현재 활성 Spec

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 `Spec` 문서다.
`ValidationCriteria`와 `Task`는 이 문서의 활성 spec을 기준으로 작성하며, GitHub Issue는 각 `Task`를 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 Spec
### ID
SPEC-0025

### 이름
운영용 조회 응답 준비 데이터 실제 SQL 보강

### 목표
- 운영용 조회 응답 대표 검증에 쓰는 예시 SQL 파일을 실제 로컬 H2 준비와 확인에 사용할 수 있는 수준으로 보강한다.
- retry 원본 실행 준비 데이터, rerun 조치 이력 준비 데이터, 실행 근거 확인 SQL을 대표 검증 절차와 직접 연결한다.
- 대표 실제 앱/H2 검증에서 준비 데이터 파일을 실제로 적용하고, 대표 결과가 준비 데이터 가이드와 모순되지 않는지 확인한다.
- 예: `retry-source-execution-seed.example.sql`로 원본 실행을 넣고 `retry-runtime-evidence-check.example.sql`로 결과 row를 확인할 수 있는 최소 실행 가능 상태를 만든다.

### 대상 문제
- 준비 데이터 이름 규칙, 파일 구조, 적용 절차 문서는 정리됐지만 예시 SQL 파일은 아직 설명용 뼈대라서 실제 로컬 H2 준비에 바로 쓸 수 없다.
- 대표 검증에서 준비 데이터 파일을 쓰려면 결국 임시 SQL을 다시 작성해야 해서, 문서와 파일 구조가 있어도 재현성이 낮다.
- schema나 enum 값이 바뀌면 예시 SQL도 같이 보강돼야 하는데, 지금은 실제 SQL이 없어 대표 검증 준비를 파일 수준에서 점검하기 어렵다.

### 범위
- 운영용 조회 응답 대표 검증에 쓰는 준비 데이터 예시 SQL 파일을 실제 SQL로 보강한다.
- 아래 내용을 최소 범위로 포함한다.
  - retry 원본 실행 준비 데이터 예시 SQL 실제 INSERT 또는 최소 실행 가능 SQL
  - rerun 조치 이력 준비 데이터 예시 SQL 실제 INSERT 또는 최소 실행 가능 SQL
  - rerun/retry 실행 근거 확인 SQL 실제 SELECT 또는 최소 확인 가능 SQL
  - 대표 실제 앱/H2 검증에서 준비 데이터 파일을 어떤 순서로 적용하는지와 실제 적용 결과 근거
- 현재 운영용 조회 응답 세트에 적용한다.
  - `rerun`
  - `retry`
  - 단건 조회
  - 목록 조회
  - 이력 조회
  - 관리자 조치 응답
- 현재 대표 검증에서 실제로 필요했던 retry 원본 실행 준비 데이터와 acknowledge/unacknowledge 조치 이력 준비 경로를 우선 대상으로 삼는다.
- 현재 준비 데이터 가이드와 준비 데이터 파일 구조를 기준으로, 문서 설명이 실제 SQL과 대표 검증까지 이어지도록 만든다.

### 비대상
- 새 조회 endpoint 추가
- 응답 필드 추가 또는 rename
- 기준 파일 자동 생성 스크립트 도입
- 대표 검증 전체 자동화
- 준비 데이터 이름 규칙과 저장 위치 규칙 재정의
- 사용자용 UI

### 외부 계약
- 기존 rerun/retry/단건 조회/목록 조회/이력 조회/관리자 조치 경로와 HTTP status 계약은 유지한다.
- 기존 응답 필드 의미는 바꾸지 않는다.
- 이번 spec은 준비 데이터 예시 SQL과 대표 검증 근거를 보강하는 작업이며, 운영 API 계약 자체를 바꾸는 작업이 아니다.

### 핵심 시나리오
1. 준비 데이터 실제 SQL 안전망 확인
   - 실제 SQL 보강을 시작하기 전에 대표 검증 회고와 자동 검증이 시작 근거로 충분한지 먼저 확인한다.
   - 부족한 경우만 최소 문서/테스트 보강을 검토한다.
2. 입력 준비 데이터 실제 SQL 보강
   - retry 원본 실행 준비 데이터와 rerun 조치 이력 준비 데이터에 실제 SQL을 넣는다.
   - schema 컬럼과 enum 값에 맞는 최소 실행 가능 상태를 만든다.
3. 실행 근거 확인 SQL과 대표 검증 절차 연결
   - 실행 근거 확인 SQL 파일에 실제 SELECT를 넣고, 어떤 대표 검증 시나리오와 연결되는지 정리한다.
   - 필요한 경우 준비 데이터 가이드 문구를 실제 SQL 단계에 맞게 최소 보정한다.
4. 실제 준비 데이터 적용 검증과 문서 마감
   - 대표 실제 앱/H2 검증에서 준비 데이터 파일을 실제로 적용하고 대표 결과를 확인한다.
   - targeted/full test를 다시 돌려 seed SQL 보강으로 기존 자동 검증 흐름이 깨지지 않았는지 확인한다.

### Task 분해 기준
- `TASK-0001` 준비 데이터 실제 SQL 안전망 확인
- `TASK-0002` 입력 준비 데이터 실제 SQL 보강
- `TASK-0003` 실행 근거 확인 SQL과 대표 검증 절차 연결
- `TASK-0004` 실제 준비 데이터 적용 검증과 문서 마감

### 연결될 ValidationCriteria
- `manual-rerun-seed-safety-net-preserved`
- `manual-rerun-seed-input-sql-executable`
- `manual-rerun-seed-evidence-sql-executable`
- `manual-rerun-seed-representative-application-verified`

### 필수 테스트 시나리오
- 준비 데이터 실제 SQL을 보강하는 동안 기존 rerun/retry/단건 조회/목록 조회/이력 조회/관리자 조치 자동 검증은 유지된다.
- 대표 검증에서 예시 SQL 파일을 실제로 적용할 수 있다.
- 대표 실제 앱/H2 검증 뒤 evidence 확인 SQL로 기대 row를 조회할 수 있다.
- seed SQL 보강 후에도 targeted test와 full test에서 기존 자동 검증 흐름이 유지된다.

## 후속 Spec 후보
### ID
SPEC-0026

### 이름
운영용 조회 응답 준비 데이터 적용 보조 명령 정리

### 시작 조건
- `현재 활성 Spec`이 완료되고, 예시 SQL 파일이 대표 검증 준비에 실제로 쓸 수 있는 수준으로 올라간 뒤 시작한다.

### 목표
- 준비 데이터 적용과 대표 검증에 반복적으로 쓰는 CLI 명령, H2 shell 조회 명령, 준비 데이터 적용 절차 보조 명령을 정리한다.

### 후속 변경 범위
- 준비 데이터 적용 보조 명령 예시 정리
- 대표 검증 반복 실행 절차 보조 문서 정리
- H2 shell 확인 명령 예시 정리

### 후속 변경 비대상
- 사용자용 UI
- 장기 저장소 도입

### 후속 검증 방향
- 문서와 보조 명령만으로 대표 검증 반복 절차를 더 빠르게 따라갈 수 있는지 확인한다.
