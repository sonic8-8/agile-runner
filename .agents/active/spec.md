# 현재 활성 Spec

## 문서 목적
이 문서는 현재 활성 구현 범위와 후속 구현 범위를 고정하는 `Spec` 문서다.
`ValidationCriteria`와 `Task`는 이 문서의 활성 spec을 기준으로 작성하며, GitHub Issue는 각 `Task`를 외부에서 추적하기 위한 수단으로만 사용한다.

## 현재 활성 Spec
### ID
SPEC-0023

### 이름
운영용 조회 응답 예시 준비 데이터 정리

### 목표
- 운영용 조회 응답 대표 검증에 필요한 준비 데이터를 새 작업자도 반복 가능하게 준비할 수 있도록 정리한다.
- 대표 검증용 준비 데이터와 문서용 기준 파일의 역할 차이를 분명히 해서, 준비 데이터를 문서 예시처럼 다루거나 기준 파일로 오해하지 않게 한다.
- 준비 데이터 이름 규칙, 저장 위치, 갱신 기준을 문서와 준비 데이터 파일로 고정해 이후 대표 검증 준비 비용을 낮춘다.

### 대상 문제
- `SPEC-0020` 이후 대표 실제 앱/H2 검증에서 retry source execution 같은 준비 데이터가 반복적으로 필요했지만, 현재는 직접 작성한 SQL이나 임시 준비에 의존하는 부분이 있다.
- 준비 데이터 이름 규칙과 저장 위치가 정리돼 있지 않아, 이후 작업자가 어떤 준비 데이터를 새로 만들고 어떤 준비 데이터를 재사용해야 하는지 추론해야 한다.
- 대표 검증 준비 과정에서 enum 값이나 runtime schema 컬럼이 현재 코드와 어긋나면 제품 버그가 아닌 거짓 실패가 먼저 발생할 수 있는데, 준비 데이터와 대표 검증 결과의 경계가 문서와 준비 데이터 파일로 충분히 고정돼 있지 않다.

### 범위
- 운영용 조회 응답 대표 검증에 필요한 준비 데이터 규칙을 문서와 준비 데이터 파일로 정리한다.
- 아래 내용을 최소 범위로 포함한다.
  - 준비 데이터 이름 규칙
  - 준비 데이터 저장 위치와 파일 단위 기준
  - 원본 실행 준비 데이터와 관리자 조치 이력 준비 데이터를 어떤 단위로 나누는지
  - 준비 데이터를 만들거나 갱신할 때 무엇을 같이 확인해야 하는지
  - 준비 데이터와 문서용 기준 파일, 대표 검증의 경계
- 현재 운영용 조회 응답 세트에 적용한다.
  - `rerun`
  - `retry`
  - `query`
  - `list`
  - `history`
  - `action`
- 현재 대표 검증에서 실제로 필요했던 retry source execution, action history, list/query 대표 row 준비 경로를 우선 대상으로 삼는다.
- 문서 규칙이 실제 현재 준비 데이터 흐름과 충돌하지 않는지 검토하고, 필요한 범위에서 준비 데이터 파일이나 관련 문서를 보정한다.

### 비대상
- 새 조회 endpoint 추가
- 응답 필드 추가 또는 rename
- 기준 파일 자동 생성 스크립트 도입
- 대표 검증 전체 자동화
- 실제 적용 명령 순서와 실행 체크리스트 상세화
- 사용자용 UI

### 외부 계약
- 기존 rerun/retry/query/list/history/action 경로와 HTTP status 계약은 유지한다.
- 기존 응답 필드 의미는 바꾸지 않는다.
- 이번 spec은 준비 데이터 운영 규칙과 대표 검증 준비 파일을 정리하는 작업이며, 운영 API 동작 자체를 바꾸는 작업이 아니다.

### 핵심 시나리오
1. 준비 데이터 규칙 안전망 확인
   - 대표 실제 앱/H2 검증에서 준비 데이터가 실제로 필요한 지점과 기존 회고 근거가 이미 충분한지 먼저 확인한다.
   - 부족한 경우만 최소 문서/테스트 보강을 검토한다.
2. 준비 데이터 이름과 저장 위치 규칙 정리
   - 어떤 대표 검증 준비 데이터가 어떤 이름 규칙으로 저장되는지 정리한다.
   - 원본 실행 준비 데이터, 관리자 조치 이력 준비 데이터, 실행 근거 준비 데이터를 어떤 파일 단위로 나누는지 고정한다.
3. 준비 데이터 갱신 기준과 대표 검증 경계 정리
   - 준비 데이터를 만들거나 갱신할 때 schema나 enum 값처럼 같이 확인해야 하는 항목을 적는다.
   - 준비 데이터와 문서용 기준 파일, 대표 검증 결과를 언제 서로 섞지 않는지 경계를 분명히 한다.
4. 준비 데이터 규칙 문서 마감과 정합성 확인
   - 새 작업자가 문서와 준비 데이터 파일만 읽고 대표 검증 준비 경로를 따라갈 수 있는지 점검한다.
   - targeted/full test를 다시 돌려 문서와 준비 데이터 규칙 정리로 기존 자동 검증 흐름이 깨지지 않았는지 확인한다.

### Task 분해 기준
- `TASK-0001` 준비 데이터 규칙 안전망 확인
- `TASK-0002` 준비 데이터 이름과 저장 위치 규칙 정리
- `TASK-0003` 준비 데이터 갱신 기준과 대표 검증 경계 정리
- `TASK-0004` 준비 데이터 규칙 문서 마감과 정합성 확인

### 연결될 ValidationCriteria
- `manual-rerun-seed-safety-net-preserved`
- `manual-rerun-seed-naming-rules-defined`
- `manual-rerun-seed-update-boundary-defined`
- `manual-rerun-seed-guide-readiness-verified`

### 필수 테스트 시나리오
- 준비 데이터 규칙을 정리하는 동안 기존 rerun/retry/query/list/history/action 자동 검증은 유지된다.
- 새 작업자가 문서만 읽고 준비 데이터 이름 규칙과 저장 위치 규칙을 이해할 수 있다.
- 준비 데이터 갱신 기준과 대표 실제 앱 검증과의 경계가 문서와 준비 데이터 파일에서 바로 읽힌다.
- 문서 정리 후에도 targeted test와 full test에서 기존 자동 검증 흐름이 유지된다.

## 후속 Spec 후보
### ID
SPEC-0024

### 이름
운영용 조회 응답 준비 데이터 적용 절차 정리

### 시작 조건
- `현재 활성 Spec`이 완료되고, 준비 데이터 이름 규칙과 갱신 경계가 문서와 파일로 고정된 뒤 시작한다.

### 목표
- 대표 실제 앱 검증을 실행할 때 준비 데이터 적용 순서와 점검 순서를 더 반복 가능하게 정리한다.

### 후속 변경 범위
- 준비 데이터 적용 명령 순서
- 준비 데이터 적용 전 schema와 enum 점검 순서
- 대표 검증 준비 체크리스트 정리

### 후속 변경 비대상
- 사용자용 UI
- 장기 저장소 도입

### 후속 검증 방향
- 대표 실제 앱 검증 준비 절차와 점검 순서가 새 작업자에게도 바로 읽힌다.
